[C | Context]
這部分定義了系統運行的環境、資料結構以及特殊的識別標記，是執行任務前的基礎知識。

* **運作環境：** 任務存在於 GitHub Issue 中，且與 Telegram 整合。
* **資料存放位置：**
    * 主要檔案：`workspaces/issue-{issue_number}/issue.md`。
    * 輔助檔案：`workspaces/issue-{N}/` 資料夾內的其餘檔案。
    * 脈絡檔案：`.memory`（記錄前次決策與摘要）。
* **系統標記定義：**
    * ``：表示此留言來自 Telegram，訊息分為「路由資料（無視）」與「實際內容（核心）」，兩者以 `---` 分隔。
    * ``：系統自動產生的執行回報。
* **排除對象：** 系統標記中的路由資料，以及所有的系統自動執行回報（`githubclaw-brain-result`）。

[O | Objective]
這部分是系統必須執行的行動邏輯，依照優先順序確保任務解讀的準確性。

### 1. 定位與解析訊息
* **搜尋最新指令：** 從 `issue.md` 中尋找「最新一則」含有 `telegram-meta` 的 comment。
* **提取原始訊息：** 僅擷取 `---` 分隔線**之後**的文字作為核心指令。

### 2. 脈絡追溯策略（若指令不完整時）
當最新指令（如：「繼續」）不足以判斷任務時，需啟動以下遞補機制：
1.  **時間倒序追蹤：** 往回讀取更早的 `telegram-meta` 留言，直到意圖完整。
2.  **全局讀取：** 若留言串仍不足，讀取整個 `issue.md`（含標題、Body 與所有留言）。
3.  **延伸參考：** 若以上皆不足，查閱 Workspace 資料夾下的其他檔案，並**優先讀取 `.memory`** 以銜接前次執行脈絡。

### 3. 判斷邏輯摘要表

| 順序 | 執行對象 | 目的 |
| :--- | :--- | :--- |
| **P1** | 最新 `telegram-meta` comment (`---` 後文字) | 獲取當前核心指令 |
| **P2** | 歷史 `telegram-meta` comments (倒序) | 補完不完整的指令意圖 |
| **P3** | `issue.md` 完整內容 | 取得 Issue 原始背景 |
| **P4** | `.memory` 及 Workspace 相關檔案 | 取得前次決策與延續工作的背景 |
